Proposal: Enabling GPU Ray Tracing using NVIDIA OptiX

Atte Ahmavaara, Kelly Liu, Lulu Yu, Alexander Yu

Summary

In our project we will be using the Nvidia OptiX ray tracing API for hardware acceleration to achieve significant speedup in our in-class path tracer. The GUI and UX will be adapted to be more interactive to allow the user to leverage the significantly faster render times. We will be tackling GPU based acceleration, which is why we decided to use Nvidia OptiX in addition to the provided framework of the course project.

Problem Description

The ray tracer we implemented in Project 3 is effective, but not efficient. The problem we intend to tackle is making the ray tracer efficient and effective. This is important as it allows us to achieve industry standard rendering, improve efficiency in ray tracing, add new capabilities to our ray tracer (like live-rendering scenes), and improve speed up by orders of magnitude. Our solution hopes to leverage NVIDIA OptiX APIs to implement optimal ray tracing performance on the GPU. Some challenges we may face include learning the API calls required and its documentation, integrating external frameworks outside of project scope. Further, almost all of the work from project 3 will need to be ported to NVIDIA OptiX due to how centrally the framework operates.

Goals and Deliverables

To showcase our results, we plan to use many of the classic scenes from class: CBbunny, CBdragon, CBlucy, CBspheres, etc. Since our goals are purely performance driven, we do not need additional/special scenes to show progress. Rather, we will be presenting both quality and performance comparisons, consisting of equal-time quality comparisons, and similar-quality time comparisons.

At a baseline, we aim to deliver a near-clone of the in-class ray tracer, but ported entirely to the GPU, inside of NVIDIA's vast OptiX engine. This will likely require a near complete rewrite of all ray tracing infrastructure of the in-class ray tracer, due to using NVIDIA's data structures and engine, including BVH construction and traversal, rather than CGL, the boiler-plate code given to us by course staff, and our previously self-implemented tasks. We would expect to see at least a single order of magnitude difference to the CPU implementation, due to the embarrassingly parallel problem that is ray tracing thousands of pixels.

Subsequent features we want to implement include changes to the GUI and UX such that the user can truly leverage and “feel” the speedup / power of the pipeline. Whereas before it would take many minutes to render a tiny change in lighting or focus, we want to be able to control it using intuitive mouse controls in near real time, at least on smaller images. Further, Nvidia OptiX includes many opportunities to use AI-based denoising and general image quality improvements on top of the core ray tracing features. If left with extra bandwidth for this project, we hope to take advantage of some of these for both additional quality and speed, likely by requiring less samples to achieve the same quality.
